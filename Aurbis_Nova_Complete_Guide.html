Project Overview

🏢 What is Aurbis Nova?

Aurbis Nova is an AI-powered workspace booking assistant for Aurbis — a premium managed office space provider across Bangalore. It uses Large Language Models (LLMs) combined with Retrieval-Augmented Generation (RAG) to let users find, compare, and book workspaces through a natural conversation interface — just like talking to a smart concierge.

What Problem Does It Solve?

Traditional workspace booking requires browsing static websites, filtering, calling support. Aurbis Nova lets users simply type: "I need a conference room for 10 people in Koramangala tomorrow at 3pm under ₹2000/hr" — and the AI handles everything else.

💬 Chat Mode

Users have a natural conversation with Nova, the AI assistant. Nova understands context, remembers previous messages, asks for missing details, and confirms bookings step by step.

🤖 Agent Mode

An autonomous AI agent that uses tools (search, compare, price, availability) in a reasoning loop to answer complex workspace questions without needing follow-up questions.

Key Capabilities

🔍 Smart Search
41 spaces · 13 Bangalore areas. Filter by location, capacity, type, price.
💬 Conversational Booking
Book through natural language. Nova collects name, date, time, email step by step.
📧 Email Confirmations
Beautiful HTML confirmation email with booking ID sent via Gmail.
⚡ ReAct Agent
Autonomous multi-step agent: Reason → Act → Observe → Answer.
📐 Goal Blueprint
Breaks complex goals into a task plan before executing — shows the full strategy.
🧠 RAG Knowledge
Answers from a ChromaDB vector store built from Aurbis's knowledge base.

AI / Infrastructure

Runs llama3 model locally on your machine. No API costs. The RAG engine uses Ollama embeddings and llama3 as the LLM. Port: 11434

🤖 OpenAI (Optional — Cloud)

If an OPENAI_API_KEY is set, uses gpt-4o-mini for chat and text-embedding-3-small for embeddings. Faster, smarter responses.

🗄️ ChromaDB (Vector DB)

Stores vectorized chunks of the Aurbis knowledge base. Enables semantic search — finding relevant docs by meaning, not just keywords. Stored at backend/chroma_db/

Uses Gmail App Password for sending beautiful HTML booking confirmation emails and agent result emails directly to the user's inbox.

Backend Deep Dive

main.py — The Entry Point

FastAPI app that wires everything together. At startup it creates a singleton AurbisRAG instance which sets up the LLM and vector store in a background thread. All routes use this singleton.

AurbisRAG (rag_engine.py)

The brain of the chat experience. Handles:

  • Document loading and chunking from knowledge_base.md
  • Building the ChromaDB vector store
  • Retrieving relevant context for each user message
  • Running the LLM with the retrieved context
  • Managing per-session conversation history
  • The entire multi-step booking collection flow
  • Self-healing: detecting low-confidence answers and retrying

AurbisAgent (agent_engine.py)

The autonomous agent. Handles:

  • ReAct loop: Thought → Action → Observation → repeat
  • 5 callable tools (search, details, price, compare, availability)
  • Streaming each step live to the frontend via SSE
  • Goal Decomposition: breaking complex goals into subtasks
  • Max 5 tool calls, then forced final answer

AI / LLM Architecture

Pattern 1 — RAG (Retrieval-Augmented Generation)

Instead of relying purely on the LLM's training data, RAG retrieves specific Aurbis information before generating an answer.

1. User Message"Find a room in Koramangala"
→
2. Embed QueryConvert to vector
→
3. ChromaDB SearchFind top-3 similar docs
↓
6. Stream to UserWord by word via SSE
←
5. LLM GeneratesAnswer using context
←
4. Build PromptContext + history + question

Pattern 2 — ReAct Agent (Reason + Act)

For complex workspace queries, the agent doesn't just answer — it reasons step by step, calls real tools, and observes results.

Goal: "Find the best 10-person room in Bellandur under ₹2000/hr"

Thought: I need to search for spaces in Bellandur with capacity ≥ 10 and price ≤ 2000
Action:  search_spaces(location="Bellandur", capacity=10, max_price=2000)
Observe: Found 2 spaces: BEL-001 (₹1,200/hr), BEL-003 (₹1,800/hr)

Thought: I should get full details on BEL-001 since it's cheapest
Action:  get_space_details(space_id="BEL-001")
Observe: Lake View Suite — 8 pax, 4K display, video conferencing, ₹1,200/hr

Thought: I have enough information for a full answer
Final Answer: "The best option is Lake View Suite in Bellandur..."

Pattern 3 — Self-Healing RAG

If Nova detects phrases like "I'm not sure" or "I don't have information" in its own response, it automatically reformulates the query and retries once before giving up.

Pattern 4 — Goal Decomposition

The Goal Blueprint mode sends the user's goal to the LLM and asks it to output a JSON task list with tool assignments. Each task is shown visually before execution.

Chat Flow

This is the step-by-step journey of a user message through the system:

  • 1
    User types a message in ChatWidget.jsx

    The frontend calls GET /api/chat/stream?message=...&session_id=.... A unique session_id is generated on first load and stored in React state.

  • 2
    FastAPI receives the request at chat_stream()

    It calls rag.chat(message, session_id) inside an async generator. The response is wrapped in a StreamingResponse with text/event-stream content type.

  • 3
    AurbisRAG detects booking intent or general question

    Keywords like "book", "reserve", "want a room" trigger the booking flow. Everything else goes to the RAG pipeline.

  • 4
    RAG pipeline: embed → retrieve → prompt → generate

    The query is embedded, top-3 relevant chunks are retrieved from ChromaDB, and a full prompt is built with context + chat history. The LLM generates a response token by token.

  • 5
    Streaming tokens arrive at the frontend

    ChatWidget.jsx listens to the SSE stream. As each chunk arrives, it appends to the assistant message in real time — the user sees the response being typed out.

  • 6
    Response rendered with react-markdown

    Nova's response may contain markdown (bold, tables, bullet lists). react-markdown with remark-gfm renders it as rich HTML inside the chat bubble.

  • 7
    Conversation history is maintained server-side

    The last 20 messages for each session_id are stored in a Python dict inside AurbisRAG.session_histories. Each new request appends to this history.

⚠️

Session persistence: Sessions are stored in-memory. They are lost when the backend restarts. For production, these should be stored in Redis or a database.

Booking Flow

When Nova detects booking intent, it enters a multi-turn data collection conversation:

  • 1
    Detect Booking Intent

    Keywords: "book", "reserve", "want to book", "wanna book", "I need a room". The _detect_booking_intent() function scans for these in the user message.

  • 2
    Extract Details from Message

    Nova uses regex extractors to pull: location (13 areas with typo tolerance), space type (conference/event/office/desk), capacity (number of people), date (tomorrow, Monday, specific dates), time (3pm, 15:00), name, email.

  • 3
    Ask for Missing Details (One at a Time)

    Nova determines what's missing and asks a single focused question. It never asks for all details at once — e.g. "Which location in Bangalore?" then "What date do you need it?"

  • 4
    Match to a Real Space

    _get_booking_option(location, space_type, capacity) finds the best matching space from the 41 in spaces.json. It uses a manual lookup table first, then falls back to the full catalog.

  • 5
    Show Booking Summary & Ask for Confirmation

    Nova displays: space name, location, date, time, capacity, price — and asks "Shall I confirm this booking?" Nothing is booked until the user says yes.

  • 6
    User Confirms → POST /api/book

    Frontend detects confirmation intent and calls /api/book with all collected details. Backend generates a unique AUR-XXXXXX booking ID.

  • 7
    Email Confirmation Sent

    send_booking_confirmation() sends a beautiful HTML email to the user's provided email via Gmail SMTP. Email contains booking ID, space details, access instructions.

✅ Required Fields

Name · Space/Location · Date · Time · Email

Nova collects all 5 before showing the confirmation summary.

🔄 Re-booking Support

If user says "same booking but in Whitefield", _detect_rebooking() triggers and pre-fills all previous details except the changed one.

Agent Mode

Agent Mode is accessed by clicking the 🤖 Agent Mode toggle in the top-right of the app. It has two sub-modes:

⚡
Run Agent (ReAct)

Type a complex goal, press Run. The agent autonomously calls tools, reasons at each step, and streams every thought and action live to the screen. You can watch the AI think in real time.

Theme: Blue

📐
Goal Blueprint (Decompose)

Type a goal, press Plan. Instead of executing immediately, Nova breaks it into 3–5 ordered subtasks with tool assignments. You see the plan before anything runs.

Theme: Teal

ReAct Loop — How It Works

StepWhat HappensUI Color
💭 ThoughtLLM reasons about what to do next (shown in real time)Blue
🔧 ActionLLM decides which tool to call and with what parametersAmber
📊 ObservationTool runs and returns data from the spaces databaseGreen
✅ Final AnswerLLM composes a complete, formatted answer for the userEmerald
⚠️ ErrorSomething went wrong (e.g. LLM timeout, unknown tool)Red

Goal Blueprint Steps

  • 1
    POST /api/agent/decompose

    Backend calls AurbisAgent.decompose_goal(goal) which sends the goal to the LLM with a special prompt asking for a JSON task array.

  • 2
    LLM outputs JSON task list

    Example: [{"id":1,"task":"Search spaces in Bellandur for 25 people","tool":"search_spaces","status":"pending"}, ...]

  • 3
    Frontend renders the BlueprintBoard

    Each task is shown as a numbered card with the assigned tool icon. Tasks animate in one by one. A progress bar shows completion state.

  • 4
    Robust Fallback

    If the LLM fails to return valid JSON, decompose_goal() falls back to keyword-based task generation (find → 2 tasks, book → 4 tasks, compare → 3 tasks).

Sharing Agent Results

After a completed agent run, the user can click Share by Email. This calls POST /api/notify which sends a formatted HTML email with the full agent result via Gmail.

Email System

The email system uses Gmail SMTP with an App Password (not the regular Gmail password). It sends two types of emails:

📧 Booking Confirmation

Triggered when a booking is confirmed via /api/book. Contains:

  • Unique Booking ID (e.g. AUR-XK9M2P)
  • Space name and location
  • Date and time
  • Price
  • Access instructions (show booking ID at reception)
  • Aurbis contact information

🤖 Agent Result Share

Triggered when user clicks Share on agent results. Contains:

  • The original goal asked
  • Full agent result text
  • Formatted as a clean HTML email
  • Branded with Aurbis Nova styling

How Gmail SMTP Works

  • 1
    Generate Gmail App Password

    In Gmail settings → Security → 2-Step Verification → App Passwords. Generate a 16-character password for "Mail".

  • 2
    Set in .env file

    GMAIL_USER=your@gmail.com and GMAIL_APP_PASSWORD=xxxx xxxx xxxx xxxx (remove spaces)

  • 3
    Python smtplib sends the email

    smtplib.SMTP_SSL("smtp.gmail.com", 465) → login → sendmail(). The email is a MIMEMultipart with both HTML and plain text parts for compatibility.

⚠️

Security note: Never use your real Gmail password. Always use an App Password. The .gitignore should include .env to prevent credentials being pushed to GitHub.
