01 What Is This Project?
🎯 Your 60-Second Pitch (memorize this!)

Aurbis Nova is an AI-powered workspace booking assistant for Bangalore. It lets users search, compare, and book co-working spaces through a conversational AI chatbot — like having a smart concierge available 24/7.

The backend is built with FastAPI (Python) and uses RAG (Retrieval-Augmented Generation) powered by LangChain + ChromaDB — so the AI always answers from real data about 41 actual spaces. It has a multi-tool ReAct AI Agent that can reason through complex multi-step requests. The frontend is React 18 + Vite. Users can also interact via WhatsApp (Twilio) or receive results by email.

Key highlight for interviews: "I built a production-grade AI system from scratch — not just calling an API. I implemented RAG, vector search, a ReAct reasoning loop, self-healing fallbacks, and real-time streaming."

03 RAG — Full Deep Dive

What is RAG?

RAG = Retrieval-Augmented Generation. Instead of asking an LLM (like ChatGPT) to answer from memory (which it might hallucinate), you give it the relevant data at the time of the question. Think of it like: "Here are the top 5 results — now answer based only on these."

User asks: "Cheapest conference room in Koramangala?" │ ▼ ┌─────────────────────────────────────────────────────────┐ │ Step 1 — EMBED the question │ │ "cheapest conference room koramangala" │ │ → convert to a vector (list of ~1536 numbers) │ │ [0.12, -0.45, 0.88, 0.03, ... ] │ └────────────────────┬────────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────────────┐ │ Step 2 — RETRIEVE similar docs from ChromaDB │ │ Find top-3 most similar document chunks │ │ → "Innovation Hub [KOR-001]: ₹1,500/hr, 12 pax..." │ │ → "The Boardroom [KOR-002]: ₹2,000/hr, 8 pax..." │ │ → "Collaboration Zone [KOR-003]: ₹800/hr, 20 pax..." │ └────────────────────┬────────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────────────┐ │ Step 3 — AUGMENT the prompt │ │ "Context: [those 3 chunks] │ │ User question: cheapest conference room in Kora...?" │ └────────────────────┬────────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────────────┐ │ Step 4 — GENERATE with LLM │ │ LLM reads context + question → produces answer │ │ "The cheapest is Collaboration Zone at ₹800/hr..." │ └─────────────────────────────────────────────────────────┘

How RAG is Built in This Project

📄 Step 1 — Load & Split Documents
# rag_engine.py — loading space data into documents
from langchain_core.documents import Document
from langchain_text_splitters import RecursiveCharacterTextSplitter

# Each space becomes a Document with text + metadata
doc = Document(
    page_content="Innovation Hub: 12 pax, ₹1500/hr, Zoom room...",
    metadata={"id": "KOR-001", "location": "Koramangala"}
)

# Splitter breaks large text into overlapping chunks
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,     # each chunk = 500 characters
    chunk_overlap=50,   # 50 chars overlap so context isn't lost
)
chunks = splitter.split_documents(documents)

Why overlap? If an answer spans a chunk boundary, the 50-char overlap means it still appears in one complete chunk.

🧮 Step 2 — Create Embeddings & Store in ChromaDB
# Convert chunks to vectors and store them
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import Chroma

embedding_model = OpenAIEmbeddings(model="text-embedding-3-small")

# This one line does everything:
# 1. Calls OpenAI API to embed each chunk
# 2. Saves vectors + text to chroma_db/ folder
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embedding_model,
    persist_directory="./chroma_db"
)

The chroma_db/ folder is like a database that stores all space information as numbers (vectors) so you can do similarity search.

🔍 Step 3 — Retrieve Relevant Context
# When user asks a question:
retriever = vectorstore.as_retriever(search_kwargs={"k": 4})
# k=4 → get 4 most relevant chunks

relevant_docs = retriever.get_relevant_documents("cheapest room Koramangala")
context = "\n\n".join(doc.page_content for doc in relevant_docs)

ChromaDB computes cosine similarity between the question vector and all stored chunk vectors, then returns the top-4 closest matches.

🔄 Step 5 — LLM Fallback Chain (3-tier)
# rag_engine.py — AurbisRAG class tries 3 modes
def _init_llm(self):
    # Try 1: OpenAI (gpt-4o-mini)
    try:
        from langchain_openai import ChatOpenAI
        self.llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.7)
        self.mode = "openai"; return
    except: pass

    # Try 2: Ollama (llama3 — local, FREE)
    try:
        from langchain_ollama import ChatOllama
        self.llm = ChatOllama(model="llama3", base_url="http://localhost:11434")
        self.mode = "ollama"; return
    except: pass

    # Fallback 3: DemoRAG — keyword-based, no LLM needed
    self.mode = "demo"

This makes the app resilient — it works even without internet or API keys.

04 AI Agent & ReAct Loop

What is a ReAct Agent?

ReAct = Reason + Act. Instead of answering directly, the AI loops through: Think about what to do → Use a tool → Look at the result → Think again → Use another tool → ... → Give final answer.

User Goal: "Find a conference room in Koramangala for 10 people under ₹2000 and tell me the price" │ ▼ ┌──────────────────────────────────────────────────────────────┐ │ Thought: I need to search for spaces first. │ │ Action: search_spaces(location="Koramangala", capacity=10, │ │ max_price=2000) │ │ Observe: "Found 3 spaces: Innovation Hub [KOR-001]..." │ └──────────────────────────────────────────────────────────────┘ │ ▼ ┌──────────────────────────────────────────────────────────────┐ │ Thought: I have the IDs. Now calculate the price. │ │ Action: calculate_price(space_id="KOR-001", hours=1) │ │ Observe: "Innovation Hub: ₹1,500/hr × 1hr = ₹1,500" │ └──────────────────────────────────────────────────────────────┘ │ ▼ ┌──────────────────────────────────────────────────────────────┐ │ Thought: I have all information to give the answer. │ │ Final Answer: "Innovation Hub fits 12 people at ₹1,500/hr"│ └──────────────────────────────────────────────────────────────┘

05 Self-Healing RAG

The Problem: Low-Confidence Answers

Sometimes the RAG retrieves irrelevant context, and the LLM gives a vague answer like "I'm not sure about that". Instead of showing this to the user, we detect it and retry with a better-formed query.

🔧 How Self-Healing Works
# rag_engine.py — chat_with_healing() method
async def chat_with_healing(self, message: str, session_id: str) -> dict:

    # Step 1: First attempt with original query
    result = await self._chat(message, session_id)

    # Step 2: Check confidence of the answer
    confidence = self._score_confidence(result["answer"])

    # Step 3: If confidence is LOW, try again with a better query
    if confidence < 0.4:
        better_query = self._reformulate(message)
        result = await self._chat(better_query, session_id)
        result["healed"] = True   # tag so UI can show "Self-healed"

    return result
📊 Confidence Scoring Logic
def _score_confidence(self, answer: str) -> float:
    """Score 0.0 to 1.0 how confident the answer sounds."""
    answer_lower = answer.lower()

    # Phrases that signal low confidence
    low_confidence_phrases = [
        "i don't know", "i'm not sure", "i cannot find",
        "no information", "not available", "unable to",
        "unfortunately", "cannot answer"
    ]

    # Phrases that signal high confidence
    high_confidence_phrases = [
        "₹", "per hour", "koramangala", "whitefield",
        "capacity", "book", "available"
    ]

    score = 0.5   # start neutral
    for phrase in low_confidence_phrases:
        if phrase in answer_lower:
            score -= 0.15
    for phrase in high_confidence_phrases:
        if phrase in answer_lower:
            score += 0.1

    return max(0.0, min(1.0, score))
🔄 Query Reformulation
def _reformulate(self, query: str) -> str:
    """Make the query more specific so retrieval is better."""
    # Add domain context to help the retriever
    return (
        f"In the context of Aurbis workspace booking in Bangalore: {query}. "
        f"Please provide specific space names, prices, and locations."
    )
💡 Interview answer: "Self-healing RAG is a pattern where the system detects low-confidence answers by scanning for uncertainty phrases, then automatically retries with a reformulated query. This improves answer quality without any human intervention."
06 Semantic Search — Vector Embeddings Explained

What is Semantic Search?

Normal search finds exact words. Semantic search finds similar meaning. So a user typing "quiet spot for focused work" will find "Focus Pod — private office, silent zone" even though none of those exact words appear.

What is a Vector Embedding?

Every piece of text gets converted to a list of ~1,536 decimal numbers (a "vector"). Similar meanings → similar numbers → close together in space.

"conference room" → [0.12, -0.45, 0.88, 0.03, ...] "meeting room" → [0.11, -0.44, 0.87, 0.04, ...] ← very close! (similar meaning) "coffee shop" → [-0.78, 0.92, -0.11, 0.67, ...] ← far away Similarity is measured with COSINE SIMILARITY: cos(θ) = (A · B) / (|A| × |B|) Range: -1 (opposite) to +1 (identical) "conference room" vs "meeting room" → cos ≈ 0.97 (very similar!)

Embedding in Code

# When indexing (done once at startup):
# "Innovation Hub [KOR-001]: 12 pax, Zoom Room certified, ₹1500/hr..."
# → embed → vector stored in ChromaDB

# When user searches:
# "cheapest room for 10 people in Koramangala"
# → embed → vector → find top-4 closest stored vectors → return those docs

# The magic: Ollama/OpenAI embedding models trained on billions of texts
# already "know" that "meeting room" ≈ "conference room"

Why This Is Better Than Keyword Search

09 Real-Time Streaming with SSE

What is SSE?

Server-Sent Events (SSE) is a way for the server to push data to the browser continuously — like a live news ticker. The browser opens one connection and the server keeps sending updates.

Frontend Backend (FastAPI) ──────── ───────────────── GET /api/agent/stream?goal=... → Opens SSE connection Agent starts ReAct loop... ← data: {"type":"thought","content":"I need to search..."} ← data: {"type":"action","tool":"search_spaces","args":{...}} ← data: {"type":"observation","content":"Found 3 spaces..."} ← data: {"type":"thought","content":"Now calculate price..."} ← data: {"type":"action","tool":"calculate_price","args":{...}} ← data: {"type":"final","content":"Innovation Hub is ₹1500/hr"} Connection closes.
🔧 SSE in FastAPI (Backend)
# main.py
@app.get("/api/agent/stream")
async def agent_stream(goal: str, session_id: Optional[str] = None):
    async def event_generator():
        agent = AurbisAgent()
        async for step in agent.stream_react(goal):
            # SSE format: "data: {json}\n\n"
            payload = json.dumps(step)
            yield f"data: {payload}\n\n"

    return StreamingResponse(
        event_generator(),
        media_type="text/event-stream",    # tells browser: this is SSE
        headers={"Cache-Control": "no-cache", "X-Accel-Buffering": "no"}
    )
⚛️ SSE in React (Frontend)
// AgentPanel.jsx — reading the SSE stream
const es = new EventSource(
    `${API_BASE}/api/agent/stream?goal=${encodeURIComponent(goal)}`
)

es.onmessage = (event) => {
    const step = JSON.parse(event.data)

    if (step.type === 'thought') {

    }
    if (step.type === 'action') {

es.onerror = () => { es.close(); setRunning(false) }

# main.py — WhatsApp webhook
SIMPLE_COMMANDS = {"hi", "hello", "hey", "start", "help", "reset", "clear"}

@app.post("/api/whatsapp")
async def whatsapp_webhook(
    background_tasks: BackgroundTasks,
    Body: str = Form(""),      # Twilio sends form-encoded data
    From: str = Form(""),      # "whatsapp:+91XXXXXXXXXX"
):
    normalized = Body.strip().lower()

    # Fast path — simple commands get instant TwiML reply
    if normalized in SIMPLE_COMMANDS:
        reply = await process_message(From, Body, rag)
        twiml = Response(
            content=f'<Response><Message>{reply}</Message></Response>',
            media_type="application/xml"
        )
        return twiml

    # Slow path — RAG query runs in background, we reply via REST API
    async def handle_slow(from_: str, body: str):
        reply = await process_message(from_, body, rag)
        send_whatsapp_message(from_, reply)   # Twilio REST API

    background_tasks.add_task(handle_slow, From, Body)
    ack = Response(
        content="<Response><Message>🔍 Nova is thinking...</Message></Response>",
        media_type="application/xml"
    )
    return ack
Why two paths? Twilio requires a response within 15 seconds. RAG queries can take 10-30s. So for complex questions we immediately acknowledge ("Nova is thinking...") and send the real answer later via REST API.
📝 Markdown → WhatsApp Format Converter
# whatsapp_bot.py — _to_wa()
def _to_wa(text: str) -> str:
    # **bold** → *bold* (WhatsApp bold format)
    text = re.sub(r'\*\*(.+?)\*\*', r'*\1*', text)
    # # Heading → *Heading* (WhatsApp doesn't have headings)
    text = re.sub(r'^#{1,4}\s+(.+)$', r'*\1*', text, flags=re.MULTILINE)
    # `code` → plain text (no code formatting in WhatsApp)
    text = re.sub(r'`(.+?)`', r'\1', text)
    # Truncate at 3500 chars
    if len(text) > 3500:
        text = text[:3497] + "..."
    return text.strip()

import smtplib
from email.mime.multipart import MIMEMultipart

def send_booking_confirmation(to_email: str, booking: dict) -> bool:

        server.send_message(msg)
    return True

14 50 Interview Questions & Answers

🤖 AI / LLM Questions

What is RAG and why did you use it instead of fine-tuning?
RAG (Retrieval-Augmented Generation) fetches relevant documents at query time and gives them to the LLM as context. I used it instead of fine-tuning because: (1) our data changes often (new spaces, prices), (2) RAG is cheaper — no GPU training, (3) it reduces hallucinations since the LLM answers from real documents, (4) I can update the knowledge base by just updating ChromaDB, no retraining needed.
Explain vector embeddings in simple terms.
A vector embedding converts text into a list of numbers that captures its meaning. Similar sentences produce similar numbers. This lets us do "meaning-based search" — searching for "quiet workspace" finds "silent private office" even without exact word matches. The model (like OpenAI's text-embedding-3-small) was trained on billions of text examples to learn these number representations.
What is cosine similarity and how is it used in your project?
Cosine similarity measures the angle between two vectors. If angle = 0°, they're identical (similarity = 1.0). If angle = 90°, they're unrelated (similarity = 0). ChromaDB uses it to find the most relevant space descriptions for a user's question. The formula is: cos(θ) = (A·B) / (|A|×|B|). It's preferred over Euclidean distance because it's not affected by vector length — only direction (meaning) matters.
What is the ReAct pattern in AI agents?
ReAct stands for Reason + Act. The LLM alternates between two steps: (1) Thought — it explains what it needs to do next; (2) Action — it calls a tool with specific parameters. After each Action, it gets an Observation (the tool result) and thinks again. This loop continues until it has enough information to give a Final Answer. It's like how a human would solve a problem step-by-step rather than guessing everything at once.
What is self-healing in your RAG system?
Self-healing means the system detects when its own answer is low-quality and automatically retries. We score the answer: if it contains phrases like "I don't know" or "not available" and lacks specific data like prices or space names, confidence drops below 0.4. We then reformulate the query by adding more context ("In the context of Aurbis workspace booking in Bangalore: ...") and call the RAG pipeline again. The improved query retrieves better documents and produces a better answer.
How does your LLM fallback chain work?
The AurbisRAG class tries three modes in order: (1) OpenAI (gpt-4o-mini) — best quality; (2) Ollama (llama3) — free, runs locally; (3) DemoRAG — pure keyword matching, no LLM needed. Each is wrapped in a try/except. If OpenAI API key is invalid or quota exceeded, it falls to Ollama. If Ollama isn't running, it falls to Demo mode. The app always works, even offline.
What is ChromaDB and why use it over a regular database?
ChromaDB is a vector database — it stores text embeddings (number vectors) and supports fast similarity search. A regular SQL database can't do semantic search — it can only match exact strings. ChromaDB stores each document as a vector and finds the k nearest vectors to a query vector. I chose ChromaDB because it's free, local (no cloud needed), file-based (just a folder), and supports the same API as Pinecone/Weaviate for easy future migration.
What is hallucination in LLMs and how does your project prevent it?
Hallucination is when an LLM confidently states false information that it invented. In our project, we prevent it with RAG: instead of asking the LLM to recall information from training data, we retrieve actual space data from ChromaDB and inject it into the prompt. The LLM is instructed to answer only based on the provided context. Additionally, the system prompt explicitly says "Never say a space is confirmed unless the booking workflow completes."
What is prompt engineering? How did you use it?
Prompt engineering is crafting the input text to the LLM to get better outputs. In this project: (1) System prompt defines Nova's personality and rules; (2) Context injection adds relevant space data to every query; (3) REACT_SYSTEM prompt enforces strict output format "Thought: ... Action: ..." for the agent; (4) DECOMPOSE_PROMPT instructs the LLM to output only valid JSON; (5) Format instructions tell the LLM to use bullet points and mention prices clearly.
What's the difference between Ollama and OpenAI?
OpenAI is a cloud API — you send your text to OpenAI's servers, pay per token, and get back a response. Ollama runs an LLM locally on your own machine — completely free, private (data never leaves your computer), but requires hardware (RAM/GPU). In this project, Ollama (llama3) is the primary option for cost and privacy; OpenAI (gpt-4o-mini) is the fallback for quality when needed.

⚙️ Backend / FastAPI Questions

Why FastAPI over Flask or Django?
FastAPI is async-first (built on Starlette), which is perfect for AI workloads that do lots of I/O (API calls, database queries). It auto-generates Swagger docs from type hints. Pydantic models validate request/response data automatically. It's 2-3x faster than Flask in benchmarks. Django is too heavy for an API-only backend.
What is async/await and why does it matter here?
Async/await is Python's way of writing non-blocking code. When an async function awaits something (like an API call), it releases the CPU to handle other requests instead of just waiting. This matters here because LLM calls take 2-10 seconds. With sync code, 10 simultaneous users would queue up. With async, FastAPI handles all 10 simultaneously on one thread — much more efficient.
What is CORS and why did you need it?
CORS (Cross-Origin Resource Sharing) is a browser security policy that blocks JavaScript from calling a different origin (domain + port). Since React runs on port 5173 and FastAPI on port 8000, the browser treats them as different origins. The CORSMiddleware in main.py adds response headers telling the browser it's safe to allow these cross-origin requests.
Explain Pydantic models in your project.
Pydantic models define the expected structure of request and response data. FastAPI uses them for automatic validation. If a POST to /api/chat comes in without a "message" field, FastAPI automatically returns a 422 Unprocessable Entity error. Models like BookingRequest define all booking fields with types — this prevents invalid data from ever reaching our business logic.
What is Server-Sent Events (SSE) and why did you choose it over WebSockets?
SSE is a one-way HTTP streaming protocol where the server sends continuous updates to the client. I chose it for the agent stream because: (1) Agent steps are one-way (server → client only); (2) SSE uses regular HTTP — simpler setup, no protocol upgrade; (3) Built-in browser auto-reconnect; (4) EventSource API is simpler than WebSocket API. WebSockets would be needed if the client also needed to send mid-stream messages.
How does the WhatsApp integration work technically?
Twilio provides a WhatsApp sandbox. When a user messages the Twilio number, Twilio makes an HTTP POST to our /api/whatsapp endpoint with form-encoded data (Body, From fields). Our handler uses a two-path system: simple commands (hi/help) reply inline with TwiML XML within the 15-second timeout. Complex RAG queries use FastAPI BackgroundTasks — we immediately return an acknowledgment TwiML, then the background task processes the query and sends the reply via Twilio REST API.
How does the email service work? Why App Password?
Python's smtplib connects to Gmail's SMTP server (port 587, TLS). We login with GMAIL_USER and GMAIL_APP_PASSWORD. A Gmail App Password is a 16-character code that allows third-party apps to access Gmail without your real password. It's scoped — if it leaks, the attacker can only send emails, not access your full account. We use MIMEMultipart to send HTML-formatted emails.

⚛️ Frontend / React Questions

Explain useState vs useRef in your project.
useState stores values that, when changed, cause the component to re-render. For example, messages state updates cause the chat to show new messages. useRef stores a value that persists across renders but doesn't trigger re-renders — used for the messagesEndRef DOM reference (scroll to bottom). Rule of thumb: if the change needs to show in the UI, use useState. If it's just a reference or counter, use useRef.
How does React component communication work in your project?
Parent-to-child: props (App.jsx passes onSwitchMode callback to AIShowcase). Child-to-parent: callback props (AIShowcase calls onSwitchMode when button clicked, App.jsx handles it). Cross-component: custom events + localStorage (SpaceCard.jsx dispatches 'aurbis_favorites_changed' event; SpaceExplorer.jsx listens for it and refreshes its state). This avoids prop drilling and global state management for simple cross-component updates.
Why useMemo for the WhatsApp URL?
The WhatsApp URL is computed from goal + result, which can be long strings. Without useMemo, it recalculates on every render — even if goal and result haven't changed. useMemo caches the computed value and only recalculates when the dependencies (goal, result) change. For expensive computations or large string operations, this improves performance.
How did you implement the real QR code?
Rather than adding a QR library, I used the free qrserver.com API: <img src="https://api.qrserver.com/v1/create-qr-code/?size=96x96&data={bookingId}&bgcolor=ffffff&color=000000" />. The bookingId is encoded in the URL. This is a good trade-off — no library bundle size increase, no maintenance, and the service is reliable for this use case.
Why localStorage for favorites and booking history?
LocalStorage is perfect for user-preference data that: (1) doesn't need server storage, (2) should persist across page refreshes, (3) is device-specific (not synced). Favorites and booking history are exactly this kind of data. The cross-component sync is handled with a custom window event (aurbis_favorites_changed) — when SpaceCard saves to localStorage, it fires the event, and SpaceExplorer listens and updates its count badge.
What is Vite and why use it instead of Create React App?
Vite is a modern frontend build tool. It uses native ES modules in development, so hot module replacement (HMR) is near-instant (under 50ms) compared to CRA's 5-30 second rebuild. Production builds use Rollup for optimized bundles. Vite is the current industry standard for React apps — CRA is officially deprecated.
How does the toast success notification work technically?
I added a showSuccessToast boolean state. When the email API call succeeds, we setToast(true) and schedule setTimeout(() => setToast(false), 3500). The toast is a fixed-position div with CSS transition on transform and opacity. When toast=false, it's translated to translateY(-80px) and opacity 0 (off-screen). When toast=true, it snaps to translateY(0) and opacity 1 with a spring cubic-bezier animation.

🏗️ Architecture Questions

How is the session/conversation history maintained?
Each chat session has a UUID. The backend stores conversation history in memory (a Python dict keyed by session_id). Each message adds to that session's history. When the RAG prompt is built, history is formatted and injected into the prompt so the LLM has context of previous messages. The session_id is generated on the frontend and sent with every message.
What happens if the LLM call fails mid-stream in the Agent?
The agent is wrapped in try/except. If the LLM call throws, the generator yields a final error step: {"type":"error","content":"Agent encountered an error: ..."}. The frontend's es.onerror closes the EventSource and sets running=false. The user sees an error state in the agent panel rather than a hung spinner.
How does the booking flow work end-to-end?
1. User chats with Nova (RAG) to find a space. 2. Nova asks for details (name, email, date, time, attendees) through conversation. 3. RAG engine detects all details are collected and returns booking_confirmed=true + booking_details. 4. Frontend shows booking confirmation with a QR code and celebration animation. 5. User can opt to confirm via POST /api/book. 6. Backend generates a booking ID, saves to booking history, sends confirmation email via Gmail SMTP. 7. Email contains the QR code, booking ID, and space details.
How would you scale this to production?
I'd make these changes: (1) Replace in-memory session store with Redis for distributed sessions; (2) Replace ChromaDB local files with Pinecone or pgvector for cloud vector storage; (3) Replace JSON file with PostgreSQL for space data; (4) Add authentication (JWT); (5) Use a CDN for the React app; (6) Containerize with Docker; (7) Add rate limiting (FastAPI + slowapi); (8) Restrict CORS to specific domains; (9) Use a job queue (Celery) for email/background tasks instead of BackgroundTasks.
What security considerations did you address?
1. API keys in .env, never in code or git (in .gitignore); 2. Gmail App Password instead of real password; 3. ast.literal_eval for safe parsing of LLM-generated tool arguments (prevents code injection); 4. CORS middleware; 5. Pydantic validation on all inputs; 6. Twilio webhook handled with form data (not raw JSON) as Twilio specifies; 7. WhatsApp message truncated to 3500 chars to prevent oversized payloads.

💡 Conceptual / Explain-It Questions

What is LangChain and what does it do in your project?
LangChain is a Python library that provides building blocks for LLM applications. In this project it handles: (1) RecursiveCharacterTextSplitter — splits long space descriptions into chunks; (2) Document class — wraps text with metadata; (3) OpenAIEmbeddings / OllamaEmbeddings — converts text to vectors; (4) Chroma vectorstore — stores/queries ChromaDB; (5) Retriever abstraction — unified interface to search docs. Without LangChain, I'd have to write all of this from scratch.
What is a text chunk and why is overlap important?
A chunk is a piece of a document, typically 500-1000 characters. We split documents into chunks because LLMs have context limits and ChromaDB searches work better on focused pieces. Overlap (e.g., 50 chars) means the last 50 characters of chunk N are the first 50 of chunk N+1. This ensures that if a key fact spans a chunk boundary, it appears complete in at least one chunk and won't be missed during retrieval.
What is the Demo mode and why did you build it?
DemoRAG is a pure keyword-based response system that requires no LLM at all. It uses if/elif matching on keywords like "conference", "koramangala", "pricing". I built it so the app works in three scenarios: no API keys, no internet, or Ollama not installed. It also makes demos more predictable — if I'm presenting and Ollama is slow, Demo mode gives instant responses. It's a graceful degradation pattern.
Explain the extract_capacity() function. Why use regex?
_extract_capacity() scans natural language for any mention of a number of people. Users say things 10 different ways: "for 5 people", "team of 10", "15 attendees", "20-seater". Regex handles all these patterns: r'\bfor\s+(\d+)\b', r'(\d+)\s*(?:people|persons|pax)', etc. Using regex is much more robust than splitting on spaces or looking for exact phrases. The function tries multiple patterns and returns the first valid match (1-500 range).
What is a typo-tolerant date extractor and how did you implement it?
_extract_date() handles common misspellings of "tomorrow": tommrow, tomorow, tmrw, tmr, tomo, tomarrow — all checked against a list. This was built from real user behavior data where people typing on mobile frequently misspell "tomorrow". The function also handles: "today/tonight/tdy", day names (Monday-Sunday), and date formats like "15 July" or "15/07". This makes the booking flow feel natural instead of failing on common typos.
How does the agent prevent infinite loops?
Two mechanisms: (1) A max_steps counter — after 5 tool calls, the agent stops and gives its best Final Answer even if not complete. (2) The REACT_SYSTEM prompt explicitly says "Maximum 5 tool calls. After 5, give your best Final Answer." This prevents a runaway agent that keeps calling tools forever, which would be expensive and slow.
What is TwiML and how does it work?
TwiML (Twilio Markup Language) is XML that tells Twilio what to do in response to a webhook. When our FastAPI endpoint returns <Response><Message>Hello!</Message></Response>, Twilio reads this XML and sends that text as a WhatsApp message to the user. It's like an instruction set for Twilio's messaging infrastructure. We set the Content-Type to "application/xml" so Twilio parses it correctly.
How does the Space Explorer filtering work?
SpaceExplorer.jsx maintains local filter state: location, type, price range, favorites toggle. It fetches all spaces from GET /api/spaces once, then filters client-side using JavaScript array methods: spaces.filter(s => !location || s.location === location).filter(s => s.pricing.hourly <= maxPrice). This is better for a small dataset (41 spaces) because it's instant and avoids a network request per filter change.
What is the singleton pattern and where is it used?
A singleton ensures only one instance of a class is created. In main.py, `rag = AurbisRAG()` is called once at startup — not inside the route handlers. This is critical because AurbisRAG initialization loads ChromaDB, initializes the LLM client, and builds the retriever — expensive operations. If we created a new AurbisRAG per request, startup would take 2-5 seconds per call. The singleton is shared across all requests.
How would you add authentication to this project?
I'd add JWT (JSON Web Token) authentication: (1) Add a POST /auth/login endpoint that returns a signed JWT; (2) Add a dependency function that verifies the JWT from the Authorization header; (3) Protect routes with `Depends(get_current_user)`. On the frontend, store the JWT in an httpOnly cookie (not localStorage, to prevent XSS). FastAPI makes this easy with the `python-jose` library and OAuth2PasswordBearer scheme.

🎯 Behavioral / Story Questions

Tell me about a technical challenge you solved in this project.
The biggest challenge was the Twilio 15-second webhook timeout. My RAG + LLM pipeline takes 10-30 seconds. If I made Twilio wait, it would throw a 15002 error and the user gets nothing. I solved this with a two-path architecture: simple commands (hi/help/reset) get an instant TwiML response. Complex questions get an immediate "Nova is thinking..." acknowledgment, while a FastAPI BackgroundTask runs the RAG query and sends the actual reply via Twilio REST API. This decouples the webhook response time from LLM processing time.
What would you do differently if starting over?
Three things: (1) Use LangChain's ConversationChain instead of manually managing session history — it handles context window limits better; (2) Use pgvector in PostgreSQL instead of ChromaDB — one fewer service, better for production; (3) Add a proper job queue (Celery + Redis) from the start instead of FastAPI BackgroundTasks — more reliable, with retry logic and monitoring.
How did you ensure the AI doesn't confirm a fake booking?
The system prompt includes a critical instruction: "Never say a space is booked, confirmed, reserved, or all set from RAG context alone. A real booking is confirmed only by the backend booking workflow after the user says yes on the final summary." The actual booking (POST /api/book) is separate from the chat. The frontend only calls that endpoint when the user explicitly confirms. This is a "human in the loop" design — the AI assists but the final confirmation is a real system action.
