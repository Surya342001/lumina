Aurbis Nova is an AI-powered workspace booking assistant built as a full-stack proof-of-concept for Aurbis, a premium managed office space provider operating across Bangalore. The system enables users to discover, compare, and book workspaces through a natural, humanized conversation — no forms, no clicks, just chat.

The project demonstrates how a modern LLM + RAG (Retrieval-Augmented Generation) architecture can replace traditional booking UIs with an intelligent concierge experience. Nova understands vague queries like "i want a room for 3 in balore", handles typos and fuzzy locations, and guides users through a complete booking flow in as few as 3 natural messages.

✅
Demo Status: Fully Functional Running live at http://localhost:5173 with Ollama llama3 RAG active (rag_mode: true). OpenAI GPT-4o-mini integration is ready and auto-activates on non-corporate networks.

Project Overview & Goals
What we built and why

Problem Statement

Traditional workspace booking systems require users to navigate complex filter UIs, choose from dropdowns, fill forms, and navigate multiple pages. The experience is transactional, not conversational. Users with quick, informal needs ("I need a room for 5 people somewhere in Bangalore tomorrow morning") are forced to translate their intent into rigid form fields.

Solution

Nova replaces the entire booking funnel with a single chat interface. The AI understands context, remembers the conversation, fills in reasonable defaults when users say "anything is fine", and presents a clean booking confirmation with a unique ID and QR code.

🧠

Natural Language Understanding

Extracts intent, capacity, location, and datetime from conversational input — typos included.

🔍

RAG-Powered Knowledge

Retrieves real space data, pricing, and policies from a ChromaDB vector store using semantic search.

📋

Guided Booking Flow

State-machine booking: name → type → location → datetime → confirm. Completes in 3–5 messages.

✨

Smart Defaults

"Any room is fine" → Nova picks the best option and explains its choice transparently.

🎨

Premium UI

Animated booking confirmation card, dual-panel layout, space gallery with live filtering.

🔌

Dual AI Backend

Auto-switches between OpenAI GPT-4o-mini and Ollama llama3 depending on availability.

AI / RAG Pipeline
How the knowledge base and LLM work together

Dual-Mode Operation

The system automatically selects the best available AI backend at startup via a background thread:

ModeBackendSpeedQualityRequires
⚡ GPT-4o-mini RAG OpenAI API ~1–2s ⭐⭐⭐⭐⭐ Valid API key + internet (non-corporate)
🦙 Ollama RAG llama3 local ~3–8s ⭐⭐⭐⭐ Ollama running on localhost:11434
🤖 Demo Mode Rule-based <50ms ⭐⭐⭐ Nothing — always available as fallback

RAG Knowledge Sources

  • knowledge_base.md — Aurbis policies, amenities, pricing tiers, FAQs, booking rules, cancellation policy, member benefits
  • spaces.json — 18 workspace listings: name, type, location, capacity, hourly/daily/monthly pricing, amenities, ideal use cases, ratings, availability flags
  • Documents are chunked (800 chars / 100 overlap) and embedded into ChromaDB with semantic vector search (k=4 nearest chunks per query)

System Prompt Design

# Nova's personality & context injected with every LLM call SYSTEM_PROMPT = """You are Nova, the AI workspace assistant for Aurbis... Personality: Warm, professional, efficient. Speak like a knowledgeable concierge who genuinely wants to find the best solution. Context from knowledge base: {context} Conversation history: {chat_history} User question: {question} Nova's response:"""

Embedding Strategy

Each space document is converted into a structured text block combining name, type, location, capacity, pricing, amenities, and ideal-for tags before embedding. This ensures semantic searches like "quiet room for focused work" correctly surface spaces tagged with "deep work" in their ideal_for field.

Booking State Machine
The conversational booking flow

All bookings use a deterministic state machine stored in AurbisRAG.booking_sessions (dict keyed by session ID). This bypasses the LLM for the booking flow — responses are instant, consistent, and never hallucinated.

StepStateNova AsksSmart Defaults
→ 1 awaiting_name "What name should I put the reservation under?" Detects name from "name is X", "I'm X", "for X"
→ 2 awaiting_type "What kind of space? Conference / Event / Office / Desk" "any/suggest" → picks Conference (or Event if 30+ people)
→ 3 awaiting_location "Which area? Koramangala / Whitefield / MG Road / Indiranagar / HSR" "anywhere/any" → picks Koramangala (flagship)
→ 4 awaiting_datetime "When do you need it?" "anytime/fine/ok" → sets Today at 3:00 PM
→ 5 confirming Shows booking summary table with auto-pick notes (📌) —
✅ confirmed "Done! Booking ID: AUR-XXXXXX. See you there! 🎉" Auto-assigns best space based on location + type + capacity

Step Skipping / Smart NLU

Users can provide multiple pieces of information in a single message. The state machine extracts all available data before prompting for what's still missing:

💡
3-Message Booking Example 1. "I want a room for 5 in koramanagla, name is Surya" → extracts name=Surya, capacity=5, location=Koramangala → asks for type
2. "tomorrow at 2pm" → extracts type=conference (default) + date=Tomorrow + time=2PM → shows summary
3. "yes" → booking confirmed with ID AUR-XXXXXX

Booking ID Format

def _generate_booking_id() -> str: suffix = "".join(random.choices( string.ascii_uppercase + string.digits, k=6 )) return f"AUR-{suffix}" # e.g. AUR-K7X9QP

NLU Capabilities
What Nova understands from natural language

Capacity Extraction

Extracts people count from diverse phrasings using regex patterns:

# All of these extract capacity correctly: "room for 3" → 3 "5 people" → 5 "team of 8" → 8 "12 pax" → 12 "seats for 20" → 20 "50-seater" → 50 "capacity of 100" → 100

Fuzzy Location Matching

Handles common misspellings and short forms:

"balore", "bengaluru", "blr" → Bangalore (general) "koramanagla", "koraman" → Koramangala "whitefild", "white field" → Whitefield "indiranager", "indranagar" → Indiranagar "hsr", "h.s.r" → HSR Layout "mg", "mgroad", "brigade" → MG Road

Name Extraction

  • "my name is Surya" → Surya
  • "I'm Rahul" / "I am Priya" → Rahul / Priya
  • "book for Ananya" / "under Vikram" → Ananya / Vikram
  • Any capitalized first word as last resort

Time Extraction

  • "3pm", "3:30 PM", "15:00" → standardized time string
  • "morning" → 10:00 AM, "afternoon" → 2:00 PM, "evening" → 6:00 PM
  • "tomorrow", "today", "Monday" → relative date resolution

Setup & Installation
How to run the project from scratch

Prerequisites

  • Python 3.11+ (tested on 3.13)
  • Node.js 18+ and npm
  • Ollama installed — brew install ollama on macOS
  • llama3 model pulled — ollama pull llama3
  • (Optional) OpenAI API key for GPT-4o-mini mode

Backend Setup

# 1. Create and activate virtual environment cd backend python3 -m venv venv source venv/bin/activate # Windows: venv\Scripts\activate # 2. Install dependencies pip install fastapi uvicorn langchain langchain-openai \ langchain-ollama langchain-community chromadb \ python-dotenv pydantic # 3. Configure environment cp .env.example .env # Edit .env: add OPENAI_API_KEY if available # 4. Start Ollama (separate terminal) ollama serve # 5. Start the backend nohup python3 -m uvicorn main:app --port 8000 --host 0.0.0.0 \ > /tmp/uvicorn.log 2>&1 &

Frontend Setup

# 1. Install dependencies cd frontend npm install # 2. Start dev server npm run dev # Opens at http://localhost:5173

Verify Everything is Working

# Check backend health curl http://127.0.0.1:8000/api/health # Expected: {"status":"ok","rag_mode":true,"backend":"ollama"} # Test Ollama directly curl -s http://localhost:11434/api/generate \ -H 'Content-Type: application/json' \ -d '{"model":"llama3","prompt":"Say hi!","stream":false}' \ | python3 -c "import sys,json; print(json.load(sys.stdin)['response'])" # Test full chat curl -s -X POST http://127.0.0.1:8000/api/chat \ -H 'Content-Type: application/json' \ -d '{"message":"i want a room for 5 in koramangala","session_id":"test1"}'
⚠️
Corporate Network Note On corporate networks using Zscaler or similar proxies, OpenAI API calls (api.openai.com) are blocked. The system automatically falls back to Ollama. To use OpenAI, connect via personal hotspot or home WiFi.
